# Requirements

## Permissions

### Task permissions
- `view` — просмотр задачи
- `edit.title` — редактирование названия
- `edit.description` — редактирование описания
- `edit.status` — редактирование статуса
- `edit.assignee` — редактирование ответственного
- `edit.labels` — редактирование меток задачи
- `edit.links` — редактирование ссылок
- `edit.relations` — редактирование связей
- `edit.blocking` — редактирование блокирующих задач
- `edit.parent` — редактирование родительской задачи
- `edit.attachments` — редактирование вложений
- `share` — выдача доступа к задаче другим пользователям

### Team permissions
- `task.create` — создание задач в команде
- `team.manage_owners` — смена владельцев команды
- `team.edit_labels` — редактирование меток в задачах команды
- `team.manage_labels` — управление каталогом командных меток
- `team.manage_filters` — редактирование командных фильтров
- `team.manage_automations` — редактирование командных автоматизаций
- Все task permissions

### Task permissions bitfield

Права задачи передаются в API-ответах как целое число. Порядок бит (младший — нулевой):

| Бит | Право |
|---|---|
| 0 | `view` |
| 1 | `edit.title` |
| 2 | `edit.description` |
| 3 | `edit.status` |
| 4 | `edit.assignee` |
| 5 | `edit.labels` |
| 6 | `edit.links` |
| 7 | `edit.relations` |
| 8 | `edit.blocking` |
| 9 | `edit.parent` |
| 10 | `edit.attachments` |
| 11 | `share` |

### Team permissions bitfield

Командные права передаются в API-ответах как целое число. Порядок бит:

| Бит | Право |
|---|---|
| 0 | `task.create` |
| 1 | `team.manage_owners` |
| 2 | `team.edit_labels` |
| 3 | `team.manage_labels` |
| 4 | `team.manage_filters` |
| 5 | `team.manage_automations` |

### Permission levels
- **Владелец** — все права
- **Команды** — командные права + права по задаче
- **Непосредственные** — только права по задаче

### Sharing rules
- Пользователь может выдать только те права, которые есть у него самого
- При шаринге выдаются только task permissions
- Цепочка шаринга сохраняется (A → B → C): отзыв права у A каскадно удаляет его у B и C

## Search sorting

При поиске задач и при выборе блокирующих, связанных, родительских и дочерних задач:
1. Сначала — полное соответствие ключа или названия запросу
2. Затем — по дате последнего открытия задачи текущим пользователем (от новых к старым)

Факт открытия фиксируется при: открытии Task Screen, открытии Task Modal, получении задачи через API.

### Сортировка
Доступна только для отображаемых колонок. Поддерживается два уровня: первичная и вторичная (применяется при равенстве первичной).

- Клик по заголовку — устанавливает первичную сортировку (или меняет направление, если уже активна); вторичная сбрасывается
- Shift+клик по другому заголовку — устанавливает вторичную сортировку
- Shift+клик по заголовку первичной — меняет её направление; вторичная не меняется
- Направление первичной обозначается стрелкой в заголовке, вторичной — стрелкой с цифрой «2»

| Колонка | Направления сортировки |
|---|---|
| Ключ | По возрастанию / убыванию номера |
| Название | А→Я / Я→А |
| Дата создания | По возрастанию / убыванию |
| Дата изменения | По возрастанию / убыванию |
| Автор создания | А→Я / Я→А (по имени) |
| Автор изменения | А→Я / Я→А (по имени) |
| Ответственный | А→Я / Я→А (по имени) |
| Команда | А→Я / Я→А (по названию) |
| Родительская задача | А→Я / Я→А (по названию) |
| Наличие вложений | Сначала есть / сначала нет |
| Наличие блокеров | Сначала есть / сначала нет |
| Наличие связей | Сначала есть / сначала нет |
| Наличие ссылок | Сначала есть / сначала нет |
| Статус | Сначала opened / сначала closed |

Колонки без сортировки: метки.

Настройки таблицы (набор колонок, порядок, сортировка) сохраняются для фильтра на бэкенде и применяются для всех пользователей с доступом к нему.

## Task labels

Задача может иметь ноль, одну или несколько меток. Метки задачи обновляются передачей полного нового массива. Пустой массив удаляет все метки.

## Permissions validation

Право `view` выдаётся автоматически при добавлении участника в команду и при шаринге задачи — даже если массив `permissions` пустой или не передан. Явно передавать `view` в массиве не требуется.

Отозвать право `view` нельзя отдельно:
- у участника команды — только через удаление участника из команды
- у получателя шаринга задачи — только через полный отзыв доступа (`DELETE /api/v1/tasks/{key}/permissions/{permissionID}`)

Массив `permissions` из невалидных значений — ошибка 400. Пустой массив допустим (будет выдан только `view`).

Применяется к эндпоинтам выдачи прав участнику команды, изменения прав участника команды и шаринга задачи.

## Dangerous operations

Следующие действия требуют подтверждения пользователя перед выполнением:
- Удаление любой сущности (задача, команда, фильтр, метка, автоматизация, вложение, ссылка и т.д.)
- Удаление участника команды

Подтверждение реализуется через Confirmation Popup.

## API Authentication

Два способа авторизации, определяются по заголовку:

- `Authorization: Bearer <access_token>` — JWT (сессия браузера)
- `X-API-Token: <pat>` — Private Access Token (скрипты, автоматизации)

Middleware проверяет: есть заголовок `X-API-Token` — ищет PAT в БД; иначе парсит JWT из `Authorization`.

## Cursor-based pagination

Все списочные эндпоинты с пагинацией используют cursor-based подход.

**Запрос:** `?cursor=<string>&limit=<number>` — `cursor` опционален (отсутствие = первая страница), `limit` опционален (дефолт: 50, максимум: 100).

**Ответ:**
```json
{
  "items": [...],
  "next_cursor": "eyJpZCI6Ii4uLiIsImNyZWF0ZWRfYXQiOiIuLi4ifQ=="
}
```

- `next_cursor` — непрозрачная base64-строка, кодирует позицию последнего элемента; `null` если страниц больше нет
- Клиент передаёт `next_cursor` как `cursor` в следующем запросе
- Курсор кодирует пару `(created_at, id)` для стабильной сортировки при вставке новых записей

## Filter RSQL mode

В RSQL-режиме фильтра пользователь вводит выражение на языке RSQL. Операторы: `==`, `!=`, `=lt=`, `=le=`, `=gt=`, `=ge=`, `=in=`, `=out=`. Логические операторы: `;` (AND), `,` (OR), `()` для группировки.

Поддерживаемые поля:

| Поле | Тип | Пример |
|---|---|---|
| `status` | enum | `status==opened` |
| `title` | text | `title==*баг*` |
| `key` | text | `key==ACME*` |
| `description` | text | `description==*urgent*` |
| `author_id` | uuid | `author_id=in=(uuid1,uuid2)` |
| `assignee_id` | uuid | `assignee_id=in=(uuid1,uuid2)` |
| `label_id` | uuid | `label_id=in=(uuid1,uuid2)` |
| `team_id` | uuid | `team_id==uuid` |
| `created_at` | datetime | `created_at=gt=2024-01-01T00:00:00Z` |
| `updated_at` | datetime | `updated_at=ge=-7d` |
| `has_links` | bool | `has_links==true` |
| `has_relation` | bool | `has_relation==true` |
| `relation_task_id` | uuid | `relation_task_id=in=(uuid1,uuid2)` |
| `is_blocked` | bool | `is_blocked==true` |
| `blocker_id` | uuid | `blocker_id==uuid` |
| `is_blocker` | bool | `is_blocker==true` |
| `blocking_id` | uuid | `blocking_id=in=(uuid1,uuid2)` |
| `has_parent` | bool | `has_parent==false` |
| `parent_id` | uuid | `parent_id=in=(uuid1,uuid2)` |
| `has_attachments` | bool | `has_attachments==true` |
| `attachment_name` | text | `attachment_name==*report*` |

При переключении режима значения критериев сбрасываются.

## Filter task events

События `filter.task_entered` и `filter.task_left` генерируются только когда изменяется задача и это влияет на её вхождение в результаты фильтра. При изменении критериев фильтра эти события не генерируются — фиксируется только `filter.updated`.

---

## Blocking tasks validation

При добавлении блокирующей задачи выполняется валидация: цепочка блокирования не может замыкаться в цикл. Если валидация не пройдена — отображается Blocking Toast, задача не добавляется.
